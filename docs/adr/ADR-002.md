# ADR-002: Storage (SQLite + filesystem artifacts + Parquet logs)

- **Status:** Accepted
- **Date:** 2026-07-07
- **Decision owner:** TrafficPulse project lead (maintainer role)
- **Supersedes:** none
- **References:** `docs/architecture-review.md` §15, §19, §25

## Context

TrafficPulse must persist confirmed events, review cases, review/audit state,
evidence artifacts, observation logs, and dataset/model registries — at
capstone scale, on a single machine, offline-first. It must support deterministic
replay of reasoning and content-addressed evidence integrity. No distributed
infrastructure is justified (spec §19; architecture §25).

This ADR records a **decision only**. It implements no database, no schema, and
adds no dependency; the storage runtime is Phase 1 work.

## Decision drivers

- Single-machine, capstone-appropriate footprint.
- Deterministic, model-free replay of the reasoning layer (architecture §15).
- Content-addressed evidence integrity (U2 `EvidenceManifest` artifact SHA-256;
  U5 `scene_config_hash`).
- File-based, git-friendly governance artifacts (U3 ontology, U4 registry).

## Considered options

- **SQLite + filesystem + Parquet (chosen).**
- A server RDBMS (PostgreSQL) — rejected: unnecessary operational weight for a
  single-machine capstone.
- A document/NoSQL or object store — rejected: no distributed requirement.

## Decision

- **SQLite (via SQLAlchemy)** owns events, cases, review states, and the audit
  index.
- **Evidence artifacts** live on the **filesystem, content-addressed** by
  SHA-256 (the manifest hash and the storage layout are the same fact).
- **Observation logs** are **Parquet**, one file per `(video, run)`, and are the
  substrate for deterministic replay.
- **Registries** are **YAML in-repo** (`registry/`, `configs/`, versioned
  `configs/scenes/`).
- **Experiment tracking** is **file-based** (config YAML in, results JSON out,
  git-tagged) — no tracking service.

## Consequences

- Deterministic replay from the observation log without a GPU/model.
- Evidence integrity is verifiable from stored hashes; compatible with U2's
  `Sha256Hex` fields and U5's `scene_config_hash`.
- No DB server to operate. SQLAlchemy/PyArrow become Phase 1 runtime
  dependencies **when the storage runtime is built** — none are added now.

## Risks and mitigations

- **SQLite write concurrency is limited** — acceptable for single-machine,
  offline-first batch processing; revisit if concurrency needs emerge.
- **Parquet requires a reader** in Phase 1 — deferred; not a Phase 0-F concern.

## Revisit triggers

- A genuine multi-machine or high-concurrency write requirement.
- Artifact volume outgrowing a single filesystem.
