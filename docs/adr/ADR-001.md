# ADR-001: Detector / licence posture (AGPL Ultralytics vs permissive-only)

- **Status:** Accepted
- **Date:** 2026-07-08
- **Decision owner:** TrafficPulse project lead (maintainer role)
- **Decision deadline (met):** before the first detector-integration unit of Phase 1
- **Supersedes:** none
- **References:** `docs/architecture-review.md` §10, §11, §27

> **History:** this ADR was **Proposed — documented-open (unresolved)** at the end
> of Phase 0-F (2026-07-07), with a named owner, a deadline set at "before the
> first detector-integration unit of Phase 1," and recorded consequences (see
> `docs/phase-0-plan.md` U6 and `docs/architecture-review.md` §11). It is now
> **resolved at that deadline** — the first detector-integration unit is the next
> gated work area. The prior open state is preserved as history and not erased.

## Context

TrafficPulse needs an object detector (and an adjacent model/tracker/OCR stack).
That choice carries a licensing posture. The architecture review (§11)
deliberately left it open because it is a genuine project decision about *future
reuse and distribution*, not a technical default — and it required re-verifying
licence facts from official sources before deciding.

Detector-independent Phase 1 work has since progressed (geometry P1-U1, synthetic
trajectories P1-U2, rule-engine core P1-U3, wrong-way reasoning P1-U4, video
ingestion P1-U5) while this ADR stayed open. The next behavioral integration area
is the detector-integration unit, which this ADR gates and therefore must now
resolve.

Five distinct licence concerns must **not** be treated as interchangeable; this
ADR keeps them separate and makes **no legal claim beyond authoritative licence
text. It is not legal advice.**

1. **Framework / source-code licence** — the detector library's own code.
2. **Model-weights licence** — pretrained weights, which may differ from the code
   and may be additionally constrained by the data they were trained on.
3. **Dataset licence** — training/eval data. A framework or model licence never
   grants rights to dataset content; datasets are governed independently by the
   U4 registry.
4. **Derived-artifact licence** — whether trained/fine-tuned weights inherit the
   framework's terms.
5. **Deployment / distribution implications** — research prototype now vs any
   future commercial or closed-source distribution.

## Decision drivers

- **Freedom to reuse/redistribute later** (and any future commercial path) — the
  core reason §11 left this open.
- **Development velocity** for a time-boxed academic capstone.
- **Bounded swap cost:** keeping the detector behind the U2 `Detection` contract
  so that changing detector later stays a localized adapter change.
- **Windows-native development / Linux CI / RTX 4060 8 GB** hardware reality.
- **Offline-first architecture** (ADR-003): no dependence on a hosted detector.
- **Honest licence hygiene:** no dataset download or use before licence review
  (U4 registry); no conflation of code, weights, and dataset rights.

## Options considered

- **Option A — Ultralytics (AGPL-3.0) for velocity.** Fastest path and the most
  tutorials; commit the repository to AGPL-3.0. Ultralytics' official licensing
  page **now confirms** (verified 2026-07-08) that *trained models/weights fall
  under AGPL-3.0 by default* and that any commercial, closed-source, SaaS,
  embedded, or privately-deployed use — including custom-trained weights —
  requires a paid Ultralytics **Enterprise** licence. This upgrades the prior
  `[K]` ("confirm before reliance") assumption to a **confirmed** fact.
- **Option B — permissive-only stack (chosen).** RT-DETR / D-FINE (Apache-2.0) +
  timm (Apache-2.0) + ByteTrack / OC-SORT (MIT) + PaddleOCR (Apache-2.0). Somewhat
  more integration effort; no AGPL coupling of the code path.
- **Option C — defer / keep open.** The prior state. Only defensible while there
  is a concrete blocker preventing a responsible decision.

## Decision

**Adopt Option B — a permissive-only detector integration posture.**

- TrafficPulse adopts a **permissive-only** detector integration posture: the
  integrated detector code path uses **permissive (Apache-2.0 / MIT) licences
  only**, with **no AGPL-licensed detector code** (Ultralytics is excluded from
  the integrated stack).
- The **initial detector integration direction is RT-DETR**, integrated from a
  **permissive source** — the official Apache-2.0 implementation
  (`lyuwenyu/RT-DETR`) or its Apache-2.0 HuggingFace Transformers port — **not**
  the RT-DETR variant redistributed inside the AGPL `ultralytics` package.
- **D-FINE (Apache-2.0) remains an alternative candidate / future benchmark**
  where a DETR-family comparison or a stronger accuracy point is warranted.
- Detector output must **adapt into the frozen U2 `Detection` contract**
  (`src/trafficpulse/contracts/detection.py`). Detector-specific types must **not
  leak** into tracking, observations, rules, events, or evidence.
- **Dataset licensing remains independently governed by the U4 dataset registry.**
  This ADR does **not** claim that a code licence determines model-weight or
  dataset rights.

Option C is explicitly **not** selected: the licence facts are now verified from
official sources, a technically supported primary candidate exists, and the U2
`Detection` seam makes the choice bounded and reversible — so there is no genuine
blocker that would justify continuing to defer.

## Rationale

- **Distribution flexibility, verified.** Option A's AGPL coupling is now
  *confirmed to extend to trained weights and private deployments* (official
  licensing page, 2026-07-08). For a project whose stated driver is "future reuse
  and distribution," that would force either full AGPL open-sourcing of any
  derived service/weights or a paid Enterprise licence. A permissive-only path
  keeps the code licence clean and leaves reuse/redistribution open.
- **RT-DETR is a technically sound primary.** It is the official CVPR-2024
  implementation (Apache-2.0), is PyTorch-based (the framework the architecture
  already anticipates), is NMS-free, is available through HuggingFace Transformers
  (Apache-2.0) — a clean permissive integration path that fits the
  adapter-behind-contract design — and its R18/R34 sizes fit 8 GB with fp16
  (architecture-review §10). This matches the Windows-dev / Linux-CI / 8 GB
  constraints.
- **Bounded, low lock-in.** The detector already sits behind the frozen U2
  `Detection` contract, so choosing RT-DETR now — and swapping to D-FINE or
  another permissive detector later — is a localized adapter change, not an
  architecture change.
- **Velocity gap is acceptable.** Option A's main advantage is speed/tutorials;
  the extra RT-DETR integration effort is modest relative to the value of an
  unencumbered licence posture and is affordable within the capstone timeline.

## Consequences

- **The detector-integration gate is LIFTED.** With ADR-001 Accepted, the first
  detector-integration unit of Phase 1 is **no longer blocked** and may proceed
  under the permissive-only posture (detector behind the U2 `Detection` contract).
- **No AGPL in the integrated code path.** The integrated detector stack must
  avoid AGPL-licensed detector code (notably the `ultralytics` package). This
  constrains dependency selection for perception going forward.
- **Weights and datasets remain separately governed.** Apache-2.0 on RT-DETR /
  D-FINE *source code* does **not** by itself grant rights to any particular
  pretrained-weight file or to the datasets those weights were trained on
  (e.g. COCO, Objects365 carry their own terms). Weight provenance and any
  fine-tuning data are reviewed via the U4 registry before use or distribution.
- **This ADR adds no dependency and no code.** It records a decision only. torch,
  torchvision, transformers, RT-DETR, D-FINE, and any detector adapter are Phase 1
  integration work, introduced by the detector-integration unit — not here.
- The project's own SPDX distribution licence is a **separate decision** from
  this detector-posture decision; this ADR does not require or pre-empt it. When
  this ADR was accepted, `pyproject.toml` declared no project `license` field.
  **Update (project-licence sync, 2026-07-08):** that separate decision has since
  been made — TrafficPulse adopts **Apache-2.0** for its own source code, now
  declared in `pyproject.toml` (PEP 639 `license` / `license-files`) and the root
  `LICENSE`. This is independent of ADR-001's detector-stack posture and does
  **not** extend to datasets or model weights, which remain separately governed.

## Risks and mitigations

- **Model-weight / pretraining-data restrictions [K].** A permissive *code*
  licence can still ship with weights that carry their own terms (e.g.
  Objects365-pretrained weights). *Mitigation:* register and review the exact
  weight file and its pretraining-data terms in the U4 registry before relying on
  or distributing it; prefer weights with clearly permissive terms; treat this as
  a per-artifact check, not a code-licence inference.
- **Accidental AGPL reintroduction.** RT-DETR is also redistributed inside the
  AGPL `ultralytics` package, and BoxMOT (AGPL) sits near the tracker choice.
  *Mitigation:* integrate RT-DETR only from its Apache-2.0 source / HF Transformers
  port; keep ByteTrack/OC-SORT (MIT) for tracking; document the permissive source
  in the detector-integration unit.
- **RT-DETR integration effort exceeds estimate.** *Mitigation:* the U2 `Detection`
  seam bounds cost; fall back to D-FINE or another permissive detector without
  touching downstream layers.
- **Licence terms may change upstream.** *Mitigation:* the revisit triggers below;
  re-verify from official sources before any distribution milestone.

## Detector abstraction boundary

- The detector is an **adapter behind the frozen U2 `Detection` contract**
  (`src/trafficpulse/contracts/detection.py`): its job is to produce `Detection`
  values (`detection_id`, `camera_id`, `frame_index`, `timestamp`, `object_class`,
  `confidence`, `bbox`, optional `source_model`).
- **No detector-specific type may leak** past that seam into tracking,
  observations, rules, events, or evidence. Those layers depend only on the
  contract, never on RT-DETR / D-FINE / any framework type.
- This seam is what makes the permissive-only posture (and any later detector
  swap) a **bounded, localized** change.

## Licence distinctions recorded (no legal certainty claimed)

- **Framework / source code** — Ultralytics **AGPL-3.0** [V]; RT-DETR
  (`lyuwenyu/RT-DETR`) **Apache-2.0** [V]; D-FINE (`Peterande/D-FINE`)
  **Apache-2.0** [V]; HuggingFace Transformers **Apache-2.0**.
- **Model weights** — Ultralytics: **AGPL-3.0 by default, confirmed** by the
  official licensing page [V, 2026-07-08]. RT-DETR / D-FINE: the repositories do
  **not** grant blanket weight rights via the code licence; weight and
  pretraining-data terms are verified per artifact [K].
- **Dataset** — governed **independently** by the U4 registry; never implied by a
  framework or weight licence.
- **Derived artifacts** — under Ultralytics/AGPL, trained weights inherit AGPL
  [V]; under the permissive stack, derived-weight terms depend on the base weights
  and fine-tuning data, reviewed per artifact [K].
- **Deployment/distribution** — permissive-only keeps the *code path* free of AGPL
  copyleft; it makes **no** claim about weight or dataset redistribution rights.

## Migration / revisit triggers

- A concrete need for a detector not available under a permissive licence
  (would require a new ADR, not an edit here).
- Any change in deployment/distribution goals (closed-source or commercial
  distribution) — re-verify weight and dataset terms first.
- Any change to the upstream framework or weights licence terms (RT-DETR, D-FINE,
  Ultralytics).
- Evidence that RT-DETR is unsuitable for the Indian fixed-camera domain, moving
  the primary direction to D-FINE or another permissive detector (bounded by the
  U2 seam; recorded as a follow-up decision).
