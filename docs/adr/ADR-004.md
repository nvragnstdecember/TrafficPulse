# ADR-004: Reprocessing and event-identity semantics

- **Status:** Proposed
- **Date:** 2026-07-07
- **Decision owner:** TrafficPulse project lead (maintainer role)
- **Supersedes:** none
- **References:** `docs/architecture-review.md` §14, §19, §27

## Context

Video may be reprocessed with a new model or config version. The evidence engine
must define how event identity behaves across runs: whether a re-detected event
reuses an id, and how duplicates across runs are reconciled. The architecture
review (§19) records the direction but leaves **cross-run deduplication semantics
undecided**, and this is allowed to remain **proposed** at the end of Phase 0-F
because no event-store runtime exists yet (U2 defines the `ConfirmedEvent`
*contract* only; there is no writer).

This ADR records a **proposed direction only**; it implements nothing.

## Decision drivers

- Immutability and auditability of confirmed events (architecture §14, §19).
- Honest handling of reprocessing (no silent overwrite of prior results).
- Deferring detail until the event-store runtime actually needs it.

## Considered options

- **Per-run immutable ids + lineage linkage (proposed).** Each run mints new
  immutable event ids (e.g. ULID); reprocessing links related events by a
  `(video, rule, time-window)` lineage tuple; cross-run dedup rule deferred.
- **Stable content-derived ids across runs** — rejected for now: would require a
  frozen dedup/identity rule the project cannot yet justify.
- **Mutable/overwritten events** — rejected: violates append-only auditability.

## Decision (proposed, not accepted)

- Event ids are **immutable per run**.
- Reprocessing **creates new event ids**, linked to prior runs by the
  `(video, rule, time-window)` lineage tuple.
- **Manifests are append-only**; no run silently overwrites another.
- The exact **cross-run deduplication rule is deferred** to when the event-store
  runtime is implemented (Phase 1).

## Consequences

- Until the dedup rule is fixed, evaluation and analytics must **account for run
  identity** to avoid double-counting events across reprocessing runs.
- `ConfirmedEvent` and `EvidenceManifest` (U2) already carry the fields needed to
  express this later (event id, rule id/version, scene-config hash, model refs);
  no contract change is required to keep this proposed.

## Risks and mitigations

- **Double-counting across runs** if dedup is applied naively. *Mitigation:* keep
  runs explicit; freeze the dedup rule in this ADR before the first reprocessing
  feature ships.

## Revisit triggers

- Implementation of the event-store runtime (Phase 1).
- The first feature that reprocesses a video with a changed model/config version.
